➤ Q5

React ne peut pas se connecter directement à MySQL parce que c’est une base de données et non une API. En plus, ce serait dangereux au niveau sécurité (exposition des identifiants). Il faut donc passer par un backend.

➤ Q6

On n’utilise pas json-server en production parce qu’il n’est pas sécurisé, il ne supporte pas beaucoup d’utilisateurs et il ne permet pas de gérer une vraie logique métier.

➤ Q7

Firebase permet une connexion directe avec React parce qu’il fournit un système complet (API + sécurité + authentification). Contrairement à MySQL, il est conçu pour être utilisé côté frontend.

➤ Q8

Pour passer en production, il faut remplacer json-server par une vraie base de données, ajouter l’authentification, sécuriser l’application et la déployer en ligne.

➤ Q9

Le risque d’utiliser des libraries comme MUI ou Bootstrap, c’est d’alourdir l’application et de dépendre des mises à jour, ce qui peut parfois casser le code.

➤ Q10

Pour une application de chat en temps réel, je choisis Firebase parce qu’il gère le temps réel facilement sans besoin de backend, contrairement à json-server.

➤ Q1

Le script ne s’exécute pas parce que React échappe automatiquement les strings dans le JSX. Le HTML est affiché comme du texte et non interprété.

➤ Q2

Le script s’exécute cette fois-ci parce que dangerouslySetInnerHTML désactive la protection de React contre les attaques XSS.

➤ Q3

Oui, on voit le header Authorization: Bearer ... dans l’onglet Network. Cela veut dire que le token est bien envoyé avec les requêtes.

➤ Q4

On stocke le token dans le state React parce que localStorage est accessible par JavaScript, donc vulnérable aux attaques XSS. Le state est plus sécurisé.

➤ Q5

Avec Redux Toolkit, on n’utilise plus switch/case ni les actions manuelles. Le code est plus simple et on peut modifier le state directement grâce à Immer.

➤ Q6

Quand on toggle la sidebar, plusieurs composants se re-rendent, même ceux qui ne changent pas. Par exemple, MainContent ne devrait pas se re-render.

➤ Q7

MainContent ne se re-render plus parce que React.memo compare les props. Si elles ne changent pas, le composant ne se met pas à jour.

➤ Q8

useMemo sert à mémoriser une valeur calculée, alors que useCallback sert à mémoriser une fonction. On utilise useCallback pour éviter de recréer une fonction à chaque render.

➤ Q10

Avant optimisation, plusieurs composants se re-rendaient inutilement. Après utilisation de React.memo, il y a moins de re-renders et les performances sont meilleures.
